## v0.4.1 Release

Cryyer v0.4.1 is live, consolidating the significant logic updates from the v0.4.0 and v0.3.0 tags. This release focuses on automation, CI/CD stability, and project structure.

### Key Features
- **Non-interactive initialization**: The `cryyer init` command now supports non-interactive execution (PR #106), enabling automated setup within CI environments and provisioning scripts.
- **Pipeline flexibility**: Introduced the `--pipeline` flag (PR #108), allowing developers to specify workflow choices between `weekly`, `release`, or `both` directly from the CLI.
- **Package Renaming**: We have transitioned to a scoped npm package. Cryyer is now published as `@atriumn/cryyer` (PR #97). Update your dependency manifests to point to the new scope.

### CI/CD & Infrastructure
- **Release Automation**: Resolved npm authentication issues in the release workflow (PR #110) and fixed tag filtering specifically for scoped packages (PR #103).
- **Workflow Reliability**: Fixed source-build logic for `send-email` and `draft-email` actions (PR #100, PR #101), ensuring version parsing remains consistent across automated drafts.
- **Release-Please Configuration**: Updated `release-please` to correctly trigger on `docs` and `chore` commits (PR #98).

### Breaking Changes
- The move to `@atriumn/cryyer` requires a manual update to your `package.json`. 
- Documentation has been audited and stale references have been removed across the repository (PR #96).

Full diffs and release assets can be found on [GitHub](https://github.com/atriumn/cryyer/releases).
